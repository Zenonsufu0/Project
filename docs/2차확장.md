## 2. 새 데이터 파일 구조

### 2.1 신설 파일

#### `classrooms.txt` — 강의실 파일
```
강의실코드,강의실명,최대수용인원
```
- 강의실코드: 전체에서 고유한 PK
- 예: `R101,공학관101호,60`

#### `offerings.txt` — 학기별 개설 파일
```
개설코드,과목코드,담당교수,정원,학년제한,학과제한
```
- **개설코드**: 해당 학기 파일 내 고유 PK (타 학기 파일과 중복 허용). 서로 다른 과목끼리도 겹치면 안 됨. 실제 건대의 '과목번호(3124 등)'에 해당
- **과목코드**: `subjects.txt`의 FK
- **학년제한**: 0 = 제한 없음, 1~4 = 해당 학년 이상만
- **학과제한**: 빈 값 = 제한 없음, 단과대/학과명 = 해당 학과만
- 분반 구분: 과목코드가 같은 레코드들이 분반 관계 (동일 과목의 분반은 과목명 자동 동일)

#### `offering_schedules.txt` — 개설-요시 매핑
```
개설코드,강의실코드,요일,시작시각,종료시각
```
- 개설코드 + 강의실코드 + 요일 + 시작시각: 복합 PK
- 하나의 개설코드에 여러 행 가능 (복수 요일/강의실)
- 강의실코드: `classrooms.txt`의 FK

#### `prerequisites.txt` — 선수과목
```
과목코드,선수과목코드
```
- 과목코드가 선수과목코드를 선행 이수 조건으로 가짐
- 하나의 과목이 여러 선수과목을 가질 경우 여러 행

### 2.2 변경 파일

#### `subjects.txt` — 과목 파일 (간소화)
```
과목코드,과목명
```
- 1차의 `courses.txt`에서 분반코드·교수·요일·시간·정원·상태 제거
- 과목 고유 정보(코드·이름)만 유지
- 상태(active/inactive)는 `offerings.txt`에서 관리

#### 기존 파일 유지
| 파일 | 변경사항 |
|------|---------|
| `students.txt` | 학년 필드 추가 (`학번,비밀번호,이름,단과대,전공,학년,상태`) |
| `admins.txt` | 변경 없음 |
| `enrollments.txt` | course_code → offering_code로 변경 (`학번,개설코드,상태`) |
| `completed_courses.txt` | 변경 없음 (과목코드 기준 유지) |
| `colleges.txt` | 변경 없음 |
| `config.txt` | 변경 없음 |

## 3. 주요 로직 변경 및 추가

### 3.1 수강신청 기간 중·이후 강의 일부 수정 허용

| 수정 항목 | 수강신청 전 | 수강신청 기간 중 | 수강신청 종료 후 |
|----------|------------|----------------|----------------|
| 정원 **증가** | ✅ | ✅ (강의실 최대수용인원 이하) | ✅ (강의실 최대수용인원 이하) |
| 정원 **감소** | ✅ | ✅ (현재 신청 인원 이상까지만) | ✅ (현재 신청 인원 이상까지만) |
| 담당교강사 변경 | ✅ | ✅ (새 교강사의 요시 충돌 없어야 함) | ✅ (동일 조건) |
| 강의실 변경 | ✅ | ✅ 큰 곳: 항상 / 작은 곳: 신청 인원 수용 가능해야 함 | ✅ (동일 조건) |
| 요시(요일·시각) 변경 | ✅ | ❌ 불가 | ❌ 불가 |

**제약 검증 로직:**
- 정원 ≤ 해당 개설의 모든 강의실 중 최소 최대수용인원
- 교강사 변경: 새 교강사가 담당하는 다른 개설코드의 요시와 시간 충돌 없어야 함
- 강의실 변경: 이동할 강의실 최대수용인원 ≥ 현재 신청 인원

### 3.2 선수과목 이수 여부 검증

수강신청 시 `prerequisites.txt`에서 해당 과목의 선수과목 목록 조회 →
학생의 `completed_courses.txt`에 모두 포함되어 있는지 확인 →
하나라도 미이수 시 신청 거부 + 미이수 선수과목 목록 출력

```
!!! 오류: 선수과목 미이수로 수강신청이 불가합니다.
  미이수 선수과목: [1001] 프로그래밍기초
```

### 3.3 학년·학과별 수강 제한

수강신청 시 개설 파일의 `학년제한`, `학과제한` 확인 →
- 학년제한 > 0이고 학생 학년 < 학년제한 → 거부
- 학과제한 비어있지 않고 학생 학과 ≠ 학과제한 → 거부

## 4. 무결성 검사 (5.5절 확장)

기존 파일들의 검사 규칙 유지 위에 추가:

| 검사 항목 | 내용 |
|----------|------|
| `classrooms.txt` 구문 | 강의실코드 형식, 최대수용인원 양의 정수 |
| `subjects.txt` 구문 | 과목코드 4자리 숫자, 과목명 비어있지 않음 |
| `offerings.txt` 구문 | 개설코드 고유성, 과목코드 FK, 정원 양의 정수, 학년제한 0~4 |
| `offering_schedules.txt` 구문 | 개설코드 FK, 강의실코드 FK, 요일/시각 형식 |
| `prerequisites.txt` 구문 | 과목코드·선수과목코드 모두 subjects.txt FK |
| 정원 ≤ 강의실 최대수용인원 | offerings + offering_schedules 참조 무결성 |
| 개설코드 고유성 | offerings.txt 내에서 중복 없음 |

## 5. 구현 우선순위 (권장 순서)

```
1단계 — 데이터 구조 재설계
  ① subjects.txt, classrooms.txt, offerings.txt, offering_schedules.txt 신설
  ② storage.py 로드/저장 함수 추가·수정
  ③ validate_integrity() 확장

2단계 — 기존 기능 이전
  ④ 강의 등록·수정·삭제·활성화를 새 개설 구조 기반으로 재작성
  ⑤ 개설 과목 조회·과목 검색 UI 업데이트

3단계 — 신규 기능
  ⑥ 수강신청 기간 중·이후 부분 수정 허용 로직
  ⑦ 선수과목 검증
  ⑧ 학년·학과 제한 검증

4단계 — 검증
  ⑨ TC 재작성 (변경된 데이터 구조 반영)
  ⑩ 통합 검사 수행
```
