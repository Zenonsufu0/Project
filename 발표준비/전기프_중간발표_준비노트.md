**③ 6.8-4/5/6 — 기이수 처리 확인 다이얼로그** `{구현: 이도현}`

##### 기이수 다이얼로그가 무엇인가

"기이수(旣已修) 처리"란 이미 다른 학교·학기에서 이수한 과목을 시스템에 등록하는 기능이다 (메뉴 4번 `기이수 과목 추가`). 이를 통해 해당 과목은 재수강 표시나 수강 제한 계산에 활용된다.

**다이얼로그가 필요한 상황**: 학생이 현재 **수강신청 중(enrolled)인 과목**을 기이수로 추가하려 할 때, 단순 등록만으로는 "수강신청 중이면서 동시에 기이수 처리된" 모순 상태가 생긴다. 기획서 6.8절은 이 경우 확인창을 띄우도록 명시:

```
수강신청 중입니다. 기이수 처리하시겠습니까? (1: 예 / 0: 아니오) >
  → 1(예): 기이수 등록 완료 + 수강신청 자동 취소
  → 0(아니오): 기이수 처리 중단
```

현재 구현(`add_completed()`)은 이 분기 없이 **수강신청 여부 무관하게 즉시 기이수 등록**한다.

##### 지금 수정하지 않는 이유 — 위험 분석

**위험 1 — TC 3개가 동일 입력 시퀀스를 공유한다**

6.8-4/5/6는 모두 `setup: 1001-01 enrolled` 상태에서 동일한 입력 `['4','1001','0','0']`을 사용한다. 기획서 기대 결과는 각각 다르지만 (확인창 표시 / yes → 이중 처리 / no → 중단), 현재 스크린샷 생성 스크립트(`tc_screenshot_gen.py`)에서 세 시나리오의 입력을 모두 다르게 바꾸고, 보고서 스크린샷 3개를 전부 교체해야 한다. 스크린샷 재생성 → 보고서 재패킹까지 1시간 이상 소요.

**위험 2 — `cancel()` 호출이 추가되면 수강신청 기간 의존성이 생긴다**

6.8-5(yes 선택) 흐름에서 기이수 등록 후 수강신청 자동 취소를 위해 내부적으로 `student_service.cancel()`을 호출해야 한다. 그런데 `cancel()`은 내부적으로 `is_registration_open()` 체크를 한다:

```python
def cancel(self, course_code, section):
    if not self.is_registration_open():           # ← 수강신청 기간 체크
        return False, "!!! 안내: 현재 수강신청 기간이 아닙니다."
    ...
```

기이수 처리(메뉴 4번)는 수강신청 기간과 무관하게 항상 가능하다. 그런데 "기이수+자동취소"를 수행하면 기이수는 성공했는데 취소만 실패하는 **반쪽 성공 상태**가 발생할 수 있다. 이 예외 처리를 추가하면 코드 복잡도가 더 올라간다.

**위험 3 — 기존 통과 TC가 영향을 받을 수 있다**

`main.py` choice "4" 블록 변경 시, 기이수 추가 흐름에 관련된 TC들(`6.8-1` 목록 조회, `6.8-2` 정상 추가)이 예상치 못한 경로를 타는지 재검증 필요하다. 특히 `student_service.timetable()` 호출이 추가되면 내부적으로 `_active_enrolled_map()` 전체를 순회하므로, 데이터 상태에 따라 동작이 달라질 수 있다.

**위험 4 — 마감까지 시간이 부족하다**

4/27 09:00 마감까지 남은 시간 기준: 구현(20~30분) + 6.8-4/5/6 시나리오 입력 수정 + 스크린샷 재생성 + 보고서 재패킹 + 검증까지 최소 1.5시간이 필요하다. 잘못 구현했을 때 롤백하고 재제출하는 버퍼가 없다.

##### 결론

①②는 `main.py` 한 곳만 수정하면 됐고 기존 TC 입력 시퀀스가 그대로 유효했다. ③은 `main.py` + 스크린샷 시나리오 3개 + `cancel()` 예외 처리까지 연쇄적으로 수정이 필요하며, 잘못되면 현재 통과하는 TC까지 깨진다. **보강사항 기재로 감점을 줄이는 것이 더 안전하고 합리적인 선택**이다.

## ✍️ 발표자료 작성 주의사항

### 분량
- 학생발표 + Q&A 합쳐 **총 20분**
- 학생발표는 **15분 이내** 권장
- Q&A 시간을 줄인다고 유리하지 않음 — 지적에 대한 **항변 기회**가 사라지는 것

### 담당자 명시 (매 페이지마다)
- 각 페이지(또는 항목)마다 담당자 이름 명시
- 예: `{기획/TC: 이찬혁, 구현/검사: 이수현}`
- **명시된 담당자가 발표·답변해야 함**

### 가독성
- 심미성은 점수 무관 — 꾸밀 필요 없음
- 단, 가독성·시인성은 중요 (글꼴, 굵기, 색 대비, 글자 크기 등)
- **핵심 내용은 반드시 텍스트로 써야 함** — 말로만 하면 감점!
- 발표 대본 수준으로 전부 쓰라는 뜻은 아님

### 제출
- **PDF 1개**, 압축 없이, eCampus "팀프로젝트" → "중간발표자료"
- **팀장(또는 작성자를 잠시 팀장 지정)만 제출 가능**

## ❓ Q&A 단계 대응

### 강사 질문에 답변
- 단순 질문 → 긴장 말고 담담히 답변
- 모르면 **모른다고 솔직히** — 아는 척이 훨씬 더 큰 문제
- 분업한 부분 질문 → 반드시 **담당 학생이 답변**

### 강사 지적에 수긍 or 항변
- 안심하고 항변 가능 — 틀린 항변 자체는 불이익 없음
- 강사가 잘못 지적하는 경우도 있음 → **차분히 논리적으로 설명**
- 단, 문서가 오해하게 써져 있어서 잘못 지적된 경우 → 문서에 감점

### 주요 지적 내용
- 기획 오류
- 구현 불일치
- 검사 누락

### 🎯 검사 방법론 관련 예상 질문 대비

| 예상 질문 | 답변 요점 |
|-----------|-----------|
| 어떤 검사를 수행했나? | 전체 통합 검사(기획서 기준 TC) — 단위 검사는 보고 대상 아님 |
| 브랜치 커버리지는? | 기획서 명시 분기 기준 100% 목표, 수행한 TC 수 |
| TC는 누가 준비했나? | 기획자/제3자/구현자 중 누가 담당했는지 파악해둘 것 |
| 예상-실제 결과가 다른 TC가 있나? | 발표자료 보강사항에 명시한 3건 (미구현 항목) |
| (좌동)으로 처리한 기준은? | 예상결과와 실제결과가 동일한 경우 — 검사 PDF에 명시된 방식 |

## 👥 동료평가 정리

### 평점 기준

| 평점 | 의미 | 감점 |
|------|------|------|
| **3** | 성실, 기여도 양호 (1인분 이상) | 없음 |
| **2** | 다소 부족했으나 같은 등급이 부당하진 않음 | -1.5점 |
| **1** | 같은 등급이면 심히 부당할 정도로 불성실 | -5점 (심각 시 -10점) |
| **0** | 사실상 미참여. F 요망 | -21점 |

> 감점은 학기 총점 100점 기준, 중간·기말 두 번

### 두 가지 경로
1. **개인별 과제 (서브)** — 강사에게만 제출 (팀원 모르게) ← **이미 마감 (4/24)**
2. **발표자료 (메인)** — 팀원 간 합의된 평점, 표지에 명시 → 이게 주로 반영됨

### 발표 중 동료평가 2점 이하 처리 절차
1. 발표·문답 후 강사가 이유 질문
2. 해당 학생에게 항변 기회
3. 강사가 최종 결정

> ⚠️ 2점 이하 있는데 강사가 넘어가려 하면 → **"동료평가 확인해주삼"** 이라고 요청!
