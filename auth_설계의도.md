# auth.py — 로그인 · 회원가입 · 입력 검증

## 역할

학생/관리자의 **인증(로그인)**과 **학생 회원가입**, 그리고 회원가입 단계에서 사용하는 **필드 검증**을 담당한다.

- 기획서 6.2, 6.3, 6.4 프롬프트 로직 대응
- 기획서 4.1 학생 필드 형식 검증

## 설계 의도

### 1. 검증을 서비스에 둔 이유
- `main.py`가 단계별 입력을 받으면서 실시간으로 "학번이 이미 존재하는지", "비밀번호 형식이 맞는지"를 피드백해야 한다 (기획서 6.3의 "재대기" 요구).
- 검증 로직을 `AuthService`에 올려두면 **같은 규칙을 관리자 등록에서도 재사용** 가능 (`admin_service.register_student`도 동일 정규식 사용).
- UI 코드(`main.py`)와 규칙 코드를 섞지 않는다.

### 2. 단계별 검증 함수를 나눠둔 이유
- `validate_student_id()`, `validate_password_format()`, `validate_password_match()`, `validate_name()` — 단계마다 독립 호출.
- 기획서 6.3은 각 필드마다 개별 오류 메시지를 요구하고("!!! 오류: 학번은 숫자 9자리이어야 합니다." 등), 잘못된 필드만 재입력하도록 지정.
- 단일 메가-검증 함수였다면 재대기 흐름이 더러워진다.

### 3. `user_type` 파라미터로 비밀번호 규칙 분기
- 기획서 4.1.2 학생 비밀번호: 영문+숫자, 6~12자, 각 최소 1자
- 기획서 4.2.2 관리자 비밀번호: 영문+숫자+특수문자(`!@#$%^&*`), 8~16자, 세 종류 각 최소 1자
- 규칙은 다르지만 호출 인터페이스는 동일하게 유지. 향후 규칙이 추가되어도 함수 시그니처 변경 없이 분기 확장 가능.

## 주요 함수

| 함수 | 역할 | 기획서 대응 |
|---|---|---|
| `validate_student_id(id)` | 학번 형식(9자리 숫자) + 중복 확인 | 4.1.1 |
| `validate_password_format(pw, user_type)` | 비밀번호 규칙 검증 | 4.1.2 / 4.2.2 |
| `validate_password_match(pw, pw2)` | 회원가입 시 비밀번호 확인 단계 | 6.3 (보조) |
| `validate_name(name)` | 한글 완성형 검증 | 4.1.3 |

| `login(id, pw)` | 3분기 실패 메시지로 로그인 | 6.4 |

## 정규식 설계

### 학생 비밀번호
```regex
(?=.*[A-Za-z])(?=.*\d)[A-Za-z0-9]{6,12}
```
- Look-ahead 2개로 "영문 최소 1" + "숫자 최소 1" 강제.
- 본체 `[A-Za-z0-9]{6,12}`은 허용 문자와 길이 제한.
- 기존 코드는 Look-ahead 없이 `[A-Za-z0-9]{6,12}`만 검사해 `"abcdef"`(숫자 없음)도 통과했다 — 기획서 위반이라 이번 PR에서 수정.

### 관리자 비밀번호
```regex
[A-Za-z0-9!@#$%^&*]{8,16}
```
- `_is_valid_admin_password`에서 추가로 세 그룹 각 최소 1자 검사를 수행.
- 기존 코드는 `[^A-Za-z0-9]`로 "모든 비영숫자"를 허용해 한글·공백도 통과했다 → 기획서 명시된 `!@#$%^&*` 8개로 제한.

### 이름
```regex
[가-힣]+
```
- 기획서 4.1.3: "한국어 완성형 글자만". 단독 자모(`ㄱ`, `ㅏ`) 금지.
- `+`로 최소 1자 요구.

## 로그인 실패 3분기

기획서 7절이 요구하는 "세 가지 원인 구분" 그대로:

```
존재하지 않는 계정 → "!!! 오류: 존재하지 않는 계정입니다."
비밀번호 불일치   → "!!! 오류: 비밀번호가 틀렸습니다."
비활성(inactive)   → "!!! 오류: 비활성화된 계정입니다."
```

반환 타입은 `(role, user, message)` — 상위 레이어에서 `role`로 분기하고 `message`는 그대로 출력.

### 학생 상태 검사를 비밀번호 검사보다 먼저 수행
기획서상 두 순서 모두 합리적이나, 비활성 사용자에게 "비밀번호 틀렸습니다"를 돌려주면 사용자가 혼란에 빠진다. `inactive` 먼저 반환해 명확한 원인 전달.

## 비의도적으로 하지 않은 것

- **비밀번호 해싱 없음**: 과제 요구사항이 평문 저장이므로 해싱 도입하지 않음 (기획서 5.2.1 예시가 평문).
- **세션 관리 없음**: `login()`은 `Student`/`Admin` 객체만 반환. 세션 유지는 `main.py`가 변수로 관리.
- **동시 로그인 차단 없음**: CLI 단일 프로세스이므로 불필요.
